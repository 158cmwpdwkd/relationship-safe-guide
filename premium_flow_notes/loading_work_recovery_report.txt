1. 수정된 파일 목록
- `app/routes_premium.py`
- `app/schemas.py`
- `app/services/premium_report.py`
- `imweb/imweb premium-survey.html`
- `imweb/premium_report.html`

2. 파일별 변경 요약
- `app/routes_premium.py` / 소규모
  `premium_entry`, `report/status`에 premium loading용 로그가 추가되었다.
  `PremiumReportStatusOut` 응답에 `state`, `message`, `reused_existing`를 채우도록 바뀌었다.
  기존 status 응답 계약이 넓어져 프론트와 함께 봐야 한다.
- `app/schemas.py` / 소규모
  premium 상태 enum에 `GENERATING`, `FAILED`가 추가되었다.
  `PremiumStateOut`, `PremiumEntryOut`, `PremiumReportStatusOut`, `SurveySubmitOut`에 `message`, `reused_existing`, `state` 필드가 추가되었다.
  API response schema 계약 변경이라 단독 배포는 주의가 필요하다.
- `app/services/premium_report.py` / 중간
  `log_premium_event()`가 추가되었고, premium 상태 계산이 `READY/FAILED/GENERATING/PROCESSING`로 세분화되었다.
  `run_premium_pipeline()`에 `reused_existing`, `ALREADY_GENERATING`, `premium.generate.*` 로그와 idempotent 성격의 재사용 분기가 들어갔다.
  생성 중 row를 재사용하도록 바뀌어서 의도는 좋지만 실제 프론트와 함께 검증이 필요하다.
- `imweb/imweb premium-survey.html` / 대규모
  의도는 `submit -> generate 직접 호출`을 제거하고 wrapper(`/premium?...`)로 바로 이동시키는 쪽이다.
  그러나 현재 파일은 인코딩/문자셋이 깨진 diff가 대량 포함되어 있고, loading 작업 외의 표시 문자열까지 광범위하게 손상된 상태로 보인다.
  지금 상태로는 롤백이 우선이다.
- `imweb/premium_report.html` / 중간~대규모
  wrapper에 `STATUS_URL`, `GENERATE_URL`, `getActivePremiumToken`, `fetchStatus`, `ensureGeneration`, `premium.wrapper.*` 로그가 추가되었다.
  token-only 진입, failed 안내, generating/status polling 분기를 붙이려는 변경이 섞여 있다.
  다만 `GENERATING`/`PROCESSING` 분기 주변이 중복 삽입된 흔적이 있어 지금 상태로는 바로 배포 위험이 있다.

3. 위험도 분류 (A/B/C)
- A. 바로 살려도 되는 변경
  현재 없음.
  이번 변경은 대부분 서로 의존적이라, 단일 파일만 안전하게 살릴 수 있는 범위가 뚜렷하지 않다.
- B. 검토 후 살릴 변경
  `app/schemas.py`
  `app/routes_premium.py`
  `app/services/premium_report.py`
  이유: loading/status/idempotency 방향은 타당하지만 API 계약과 상태 의미가 바뀌므로 세 파일을 묶어서 검토해야 한다.
- C. 되돌리는 것이 나은 변경
  `imweb/imweb premium-survey.html`
  `imweb/premium_report.html`
  이유: 프론트는 현재 부분 적용 상태이며, 특히 `premium-survey`는 인코딩 손상 diff가 커서 기존 기능까지 깨질 위험이 높다.

4. 지금 바로 커밋해도 되는 최소 범위
- 권장 없음.
- 굳이 남긴다면 백엔드 3개(`app/schemas.py`, `app/routes_premium.py`, `app/services/premium_report.py`)를 한 묶음으로 별도 검토 후 커밋하는 방식이 최소 범위다.
- 프론트 2개가 빠진 상태에서 바로 배포하는 것은 loading UX 목적을 달성하지 못한다.

5. 되돌리는 것이 좋은 범위
- 우선순위 1: `imweb/imweb premium-survey.html`
  대규모 diff이며 loading/polling과 무관한 표시 문자열까지 깨진 흔적이 있다.
- 우선순위 2: `imweb/premium_report.html`
  wrapper 로직이 부분적으로만 바뀌었고 `GENERATING` 분기 중복 삽입 등 정리되지 않은 상태가 남아 있다.
- 필요 시 우선 프론트 2개만 롤백하고, 백엔드 3개는 별도 브랜치에서 다시 검토하는 편이 안전하다.

6. 다음 작업 추천
- 1. 프론트 2개(`premium-survey`, `premium_report`)를 먼저 원상 복구하거나 별도 브랜치로 분리한다.
- 2. 백엔드 3개만 남긴 상태에서 `status schema 변경`과 `generate idempotency`를 독립적으로 재검토한다.
- 3. 그 다음 라운드에서 프론트는 `wrapper 진입`, `status polling`, `token-only 진입`, `failed UI`를 처음부터 작게 다시 붙인다.
- 4. 특히 `premium-survey.html`은 인코딩이 깨지지 않는 편집 경로부터 확인한 뒤 다시 작업하는 것이 좋다.
