1. 수정한 파일 목록
- `imweb/imweb payment-fail.html`
- `imweb/imweb free-report.html`

2. 수정한 함수 목록
- `payment-fail.html`
  - `buildFreeUrlFromToken(token)`
  - `validateCanonicalFreeUrl(rawUrl)`
  - `resolveReturnUrl()`
- `free-report.html`
  - `handleSingleReloadAfterPaymentFail()`

3. payment-fail 최종 URL 선택 규칙
- 1순위: query `free_return_url` 가 유효하면 그대로 사용
- 2순위: query `free_token` 이 있으면 `/report?token=...`
- 3순위: storage free token 이 있으면 `/report?token=...`
- 4순위: `HOME_URL`

4. free-report 1회 강제 재로드 동작 방식
- payment-fail 버튼 클릭 직전에 `sessionStorage.rl_return_from_payment_fail = 1` 저장
- free-report 는 boot 전에 이 플래그 또는 `document.referrer` 의 `payment-fail` 흔적을 확인
- 아직 `rl_free_report_reload_done` 가 없으면 1회만 `window.location.replace(currentUrl)` 수행
- 두 번째 로드에서는 가드를 정리하고 그대로 정상 boot 진행

5. 추가하지 않도록 정리한 파라미터 목록
- `from_payment_fail`
- `force_reload`
- 기타 payment-fail 보정용 query

6. 테스트 시나리오별 결과
- 케이스 1: `free_return_url` 존재
  - 버튼 href 는 해당 URL 그대로 설정
  - 추가 파라미터 없이 이동
- 케이스 2: `free_token` 만 존재
  - `/report?token=...` 만 생성
- 케이스 3: payment-fail -> free-report
  - 첫 진입 시 1회만 재로드 보정
  - 이후 반복 reload 없음
- 케이스 4: 직접 `/report?token=...` 진입
  - 강제 reload 없음
- 케이스 5: 반복 새로고침
  - `rl_free_report_reload_done` 가드로 무한 루프 방지

7. 남은 리스크 1~2개
- 아임웹 바깥 섹션/위젯 렌더링 방식이 특이하면 free-report 첫 렌더 타이밍 이슈가 완전히 사라지지 않을 수 있습니다.
- free-report 의 단일 boot 구조 자체가 느린 환경에서는 추가적인 서버 응답 지연과 구분해서 확인이 필요합니다.
