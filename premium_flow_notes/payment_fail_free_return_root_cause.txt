1. payment-fail 버튼이 기존에 잘못 가리키던 실제 경로
- canonical free return URL 이 항상 명시적으로 저장되지 않아 token fallback 이 우선되거나, 저장값이 비어 있을 때 `/report?token=...` 재조합 경로로 떨어질 수 있었다.
- 이 경우 사용자가 기대한 "원래 free wrapper URL" 이 아니라 token 기반 재진입 경로로 돌아가면서 premium 상태 오염이 다시 노출될 여지가 있었다.

2. canonical free return URL을 어떻게 저장하도록 바꿨는지
- free wrapper 진입 시 현재 free 페이지 URL과 free token 으로 canonical URL 을 만든다.
- 형식:
  - current origin + current pathname + ?token={free_token}
- 이 값을 localStorage 와 sessionStorage 의 rl_free_report_return_url 에 함께 저장한다.
- 관련 로그:
  - free.return_url.capture
  - free.return_url.normalized
  - free.return_url.saved

3. free token / premium token 혼선 원인
- free 복귀가 보장되지 않을 때 rl_free_report_token fallback 과 premium 진입 상태가 섞여, 사용자가 기대한 free wrapper 재진입 대신 premium 쪽 상태가 다시 개입할 수 있었다.
- 이번 수정으로 payment-fail 은 rl_premium_report_token 을 복귀 경로 계산에 전혀 사용하지 않는다.

4. free 재진입 cleanup에서 추가로 지운 상태값
- localStorage / sessionStorage:
  - rl_premium_report_token
  - rl_premium_entry_token
  - rl_premium_last_state
  - rl_premium_runtime_state
  - rl_premium_frame_session
- payment last state:
  - rl_payment_last_mode
  - rl_payment_last_code
  - rl_payment_last_msg
- 유지:
  - rl_free_report_return_url
  - rl_free_report_token

5. 수정한 파일 목록
- imweb/imweb free-report.html
- imweb/imweb payment-fail.html

6. 테스트 시나리오
- free -> premium -> 결제 취소 -> payment-fail
  - payment-fail 진입 후 로그에서 payment_fail.return_path.raw_url, payment_fail.return_path.raw_token, payment_fail.return_path.selected 확인
- payment-fail -> 무료리포트 다시 보기
  - 버튼 href 와 click 로그가 같은 canonical free URL 을 가리키는지 확인
- 복귀 후 free report 정상 렌더
  - free.enter, free.cleanup.start, free.cleanup.removed, free.return.enter 로그 확인
- old/new token 각각에서 premium NOT_PAID 재유입 여부 확인
  - free 재진입 후 rl_premium_report_token 이 다시 생기지 않는지 확인
