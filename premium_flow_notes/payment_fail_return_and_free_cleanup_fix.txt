1. payment-fail 버튼이 기존에 제대로 작동하지 않던 직접 원인
- `payment-fail` 에 free 복귀 정보가 query 로 이미 들어와 있었지만, 선택/검증 로직이 여러 fallback 과 섞여 있어서 실제 정답 URL 이 CTA 에 바로 박히지 않았다.
- 그 결과 storage/home fallback 이 개입하거나, free 복귀 후 fail 흔적 제거가 충분히 강하지 않아 화면 오염이 남을 수 있었다.

2. query free_return_url을 어떻게 단순화해서 사용하도록 바꿨는지
- `free_return_url` query 를 최우선으로 평가
- valid 조건:
  - hostname 이 `reconnectlab.co.kr` 또는 same-origin
  - pathname 이 `/report`
  - token 존재
- 위 조건 만족 시 storage/fallback 로직보다 먼저 selected URL 로 채택
- 관련 로그:
  - payment_fail.return_path.query_url_raw
  - payment_fail.return_path.query_url_selected
  - payment_fail.return_path.query_url_rejected

3. CTA href에 실제로 어떤 URL을 넣도록 바꿨는지
- `free_return_url` 또는 fallback 으로 선택된 free URL 에 `from_payment_fail=1` 을 붙인 URL
- 예:
  - `https://reconnectlab.co.kr/report?token=...&from_payment_fail=1`
- 렌더 시점에 `returnBtn.href` 에 바로 세팅

4. free-report에서 from_payment_fail 기반으로 어떤 DOM을 제거하도록 바꿨는지
- `from_payment_fail=1` 감지 시:
  - fail heading 텍스트
  - `무료리포트 다시 보기` 버튼 DOM
  - fail card wrapper 성격의 `.page .card`/`.page` 일부
  - `결제는 완료되지 않았어요` 텍스트가 포함된 fail 조각
- 단, `#rcl-main`, `#rcl-nav`, `#rcl-footer` 내부의 free report 메인 루트는 제거하지 않음
- cleanup 후 `history.replaceState` 로 `from_payment_fail=1` 만 주소에서 제거

5. 수정한 파일 목록
- imweb/imweb payment-fail.html
- imweb/imweb free-report.html

6. 테스트 시나리오
- 결제 취소 -> payment-fail 진입
- `payment_fail.return_cta.href` 확인
- 무료리포트 다시 보기 클릭
- `/report?token=...&from_payment_fail=1` 진입 확인
- free 오염 제거 후 정상 렌더 확인
- `history.replaceState` 로 clean URL 정리 확인
