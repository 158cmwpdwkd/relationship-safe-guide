3. 변경 전 구조 요약
- free token, sid, order_id, 기존 reports.report_token 이 premium 흐름에서 섞여 있었다.
- Order는 sid 기준 여러 개 생길 수 있었지만 PaidSurvey/Report는 sid 기준 단일 row였다.
- premium read/entry 일부 경로는 order_id 없이 sid 기준 추정 조회를 했다.
- premium-survey 프론트 submit endpoint가 백엔드 라우트와 불일치했다.
- free report도 `/r/{token}`를 써서 free token과 premium token의 read 경로가 섞여 있었다.

4. 변경 후 구조 요약
- free report는 기존 `reports` 테이블과 free token으로만 유지한다.
- premium은 `Order -> PaidSurvey(order_id PK) -> PremiumReport(order_id PK)`로 분리했다.
- premium source of truth는 `order_id` 하나로 통일했다.
- premium report read는 `/r/{premium_report_token}`만 사용한다.
- free report read는 `/report?token=<free_token>`만 사용한다.
- premium entry/access-check/status/generate는 모두 `orderId/order_id` 기반으로만 동작한다.
- generate는 같은 order에 대해 idempotent 하게 기존 premium_report_token을 재사용한다.

5. source of truth 를 order_id 로 정한 이유
- 결제 성공 이후 premium 흐름은 “특정 주문”에 귀속되어야 한다.
- sid는 무료 설문 세션 식별자라서 같은 사용자가 여러 번 결제할 때 premium 흐름 분리에 적합하지 않다.
- free token은 무료 리포트 진입용 식별자이지, premium 저장/조회 식별자로 쓰면 충돌이 생긴다.
- order_id는 결제 단위로 유일하고, 재결제/재시도 상황에서도 각 premium 결과를 독립적으로 분리하기 쉽다.

6. 모델 관계 변경 사항
- UserSession 1 : N Order
  - `UserSession.orders`로 수정
  - 기존 `order uselist=False` 제거
- Order
  - `free_report_token`, `payment_method`, `created_at`, `updated_at` 추가
  - 같은 sid에 여러 order 허용 유지
- PaidSurvey
  - 기존 sid 단일 PK 모델을 폐기하고 `paid_surveys_v2` 테이블로 교체
  - `order_id` PK/FK, `sid`는 보조 FK
  - order 기준 1:1 보장
- PremiumReport
  - 신규 `premium_reports` 테이블 추가
  - `order_id` PK/FK
  - `premium_report_token` unique
  - `/r/{premium_report_token}`로 정확히 1행 조회 가능
- Report
  - free report 전용으로 유지
  - free token은 여기에만 남김

10. 남은 리스크
- 기존 운영 DB에 이미 존재하는 legacy `paid_surveys` 데이터를 `paid_surveys_v2`로 이관하는 작업은 별도 마이그레이션이 필요하다.
- `orders`에 신규 컬럼을 runtime `ALTER TABLE`로 보강하도록 넣었지만, 운영 DB 권한/DDL 제약은 배포 전 확인이 필요하다.
- admin/운영 도구는 아직 `premium_reports`를 직접 보여주지 않는다.
- premium wrapper는 orderId 기반으로 바뀌었지만, 실제 imweb 페이지 링크가 이 wrapper를 어떤 경로로 쓰는지는 최종 배포 페이지에서 재확인해야 한다.

11. 추후 운영 시 주의점
- 정책 선택: A. 재결제 허용, order별 결과 완전 분리
- 선택 이유
  - 기존 문제의 본질은 재결제 자체가 아니라 식별자 축 혼용이었다.
  - 운영 중 결제 재시도/중복 결제를 완전히 막으려면 결제 정책 예외 처리가 더 많이 필요하다.
  - order별 분리를 명확히 해두면 수동 정리 없이도 각 주문을 독립적으로 추적할 수 있다.
- 배포 전 필수 확인
  - runtime schema sync가 실제 운영 DB에서 성공하는지
  - legacy 데이터가 남아 있어도 새 테이블 경로로 정상 동작하는지
  - imweb 연결 페이지가 `orderId`를 유지하는지
  - 결제 성공 후 premium-survey, generate, `/r/{premium_report_token}`까지 E2E 수동 확인

- 결론:
- 직접 원인: premium 데이터가 sid/free token 축에 묶여 있어 다중 order를 분리하지 못한 구조
- 보조 원인: 프론트/백엔드 endpoint 불일치와 free/premium read 경로 혼용
- 가장 먼저 검증할 사용자 시나리오: 같은 무료리포트에서 order1 결제 실패 후 order2 결제 성공 -> premium-survey 제출 -> `/r/{premium_report_token}` 진입 -> 다시 order1/order2를 각각 조회해 서로의 PaidSurvey/PremiumReport가 섞이지 않는지 확인
