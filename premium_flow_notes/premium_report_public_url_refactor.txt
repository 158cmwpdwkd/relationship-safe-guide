- app/services/premium_report.py
- app/routes_premium.py
- imweb/imweb premium-survey.html

2. 수정한 함수 목록
- app/services/premium_report.py
  - build_public_premium_report_url()
  - resolve_premium_state()
- app/routes_premium.py
  - generate_premium_report()
  - finalize_premium_report()
- imweb/imweb premium-survey.html

3. 현재 onrender 도메인이 노출되던 직접 원인
- premium report generate 이후 프론트가 `window.location.href = API_BASE + "/r/" + premium_report_token` 방식으로 직접 이동하고 있었다.
- 동시에 백엔드 일부 응답도 `report_url = "/r/{token}"` 형태를 내려주고 있었다.
- 여기서 `API_BASE`가 `https://relationship-safe-guide-api.onrender.com` 이므로, 최종 사용자 주소창이 API 서버 도메인으로 바뀌었다.

4. 최종 public URL 정책
- premium report의 사용자 공개 URL은 public site wrapper를 사용한다.
- 최종 패턴:
  - `https://reconnectlab.co.kr/premium?token=<premium_report_token>`
- API 서버의 `/r/{premium_report_token}`는 wrapper 내부 iframe이 불러오는 내부 렌더 경로로만 사용한다.
- 사용자는 더 이상 API 도메인 `/r/{token}`로 직접 이동하지 않는다.

5. API base URL / public site URL 역할 분리 내용
- `API_BASE_URL` / 프론트 `API_BASE`
  - API 호출용
  - 예: premium entry, access-check, generate, internal iframe src
- `PUBLIC_SITE_URL`
  - 사용자 공개 사이트 도메인
  - 예: `https://reconnectlab.co.kr`
- `PREMIUM_PUBLIC_PATH`
  - premium wrapper public path
  - 기본값 `/premium`
- 최종 premium public URL은 항상 `PUBLIC_SITE_URL + PREMIUM_PUBLIC_PATH + ?token=...`로 생성한다.

6. wrapper 경로 설명
- 이미 `imweb/premium_report.html`이 wrapper 역할을 하고 있었다.
- wrapper는 query의 `token` 또는 `orderId`를 읽고, 내부에서 API `/r/{premium_report_token}`를 iframe으로 로드한다.
- 따라서 새 구조는 wrapper를 새로 만드는 것이 아니라, generate/access-check/state 응답과 프론트 redirect를 이 wrapper로 통일하는 방식이다.

7. 테스트 결과
- premium report generate 성공 후 응답의 `report_url`
  - 이제 `/r/{token}`가 아니라 `https://reconnectlab.co.kr/premium?token=...`를 내려주도록 수정했다.
- premium-survey 결과 보기 클릭
  - 이제 `generateData.report_url` 우선 사용
  - 백엔드 값이 없더라도 fallback은 `https://reconnectlab.co.kr/premium?token=...`
- wrapper 페이지 로드
  - 기존 wrapper는 여전히 내부 iframe src를 `API_BASE + "/r/" + token`으로 구성하므로 렌더 구조는 유지된다.
- 새로고침
  - wrapper는 query token 기반이라 `premium?token=...` 상태에서 새로고침해도 같은 report를 다시 로드한다.
- free report 경로와 혼동 방지
  - free: `/report?token=<free_token>`
  - premium: `/premium?token=<premium_report_token>`
  - 두 토큰과 진입 경로를 분리 유지한다.

8. 남은 리스크
- 실제 운영 imweb 라우팅에서 `premium_report.html`이 정말 `/premium`으로 매핑되어 있어야 한다. 현재 코드는 그 경로를 전제로 한다.
- 다른 프론트 파일에서 여전히 `API_BASE + "/r/"`를 직접 조립하는 코드가 추가로 생기면 다시 같은 문제가 생길 수 있다.
- 이번 수정은 premium 흐름 중심이다. free report 쪽 public URL 정책과 섞이지 않도록 운영 배포 시 path 매핑을 다시 확인해야 한다.

- 결론: premium report 공개 URL을 API 렌더 URL이 아니라 public site wrapper URL로 통일해서, 사용자 주소창이 `reconnectlab.co.kr`에 머물도록 수정했다.
- 직접 원인: premium-survey와 일부 backend response가 premium report URL을 `/r/{token}` 또는 `API_BASE + /r/{token}` 기준으로 사용하던 구조.
- 보조 원인: API base URL과 public site URL의 역할이 코드상 명확히 분리되지 않아, 내부 렌더 경로가 그대로 사용자 공개 URL이 된 점.
- 최종 사용자가 보게 될 premium report URL 패턴: `https://reconnectlab.co.kr/premium?token=<premium_report_token>`
