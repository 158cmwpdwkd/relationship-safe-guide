1. debug panel 활성화 방법

- premium wrapper URL에 `?debug=1`을 붙여 접속한다.
- 예:
  - `/premium_report.html?token=...&debug=1`
- debug가 켜지면 페이지 우측 하단에 `RCL Debug Buffer` 패널이 표시된다.

2. 로그 export 방법

- 패널의 `Copy` 버튼으로 최근 로그를 바로 복사할 수 있다.
- 패널의 `Clear` 버튼으로 현재 버퍼를 비울 수 있다.
- 개발자 콘솔에서 아래 함수를 직접 호출해도 된다.
  - `window.__RCL_EXPORT_DEBUG__()`
- 기본 export는 최근 300줄 기준 문자열이다.

3. wrapper와 renderer 로그가 어떻게 합쳐지는지

- wrapper 로그
  - `window.__RCL_DEBUG_LOGS__` 버퍼에 직접 적재
  - `scope: "wrapper"`

- renderer 로그
  - parent로 `postMessage({ type: "RCL_DEBUG_LOG", ... })` 전송
  - wrapper가 이를 받아 같은 버퍼에 적재
  - `scope: "renderer"`

- 결과적으로 debug panel 하나에서 wrapper + renderer 로그를 시간순으로 함께 볼 수 있다.

4. 사용자가 나에게 전달해야 할 핵심 로그 순서

1. `frame.session.start`
2. `showFrame`
3. renderer 쪽 `source.fire`
4. renderer 쪽 `sendHeight` 또는 `ignoreHeight`
5. wrapper 쪽 `message.receive`
6. wrapper 쪽 `message.target`
7. wrapper 쪽 `height.defer`
8. wrapper 쪽 `height.apply` 또는 `height.ignore`

가능하면 위 구간이 반복되는 20~40줄 정도를 그대로 복사해서 전달하면 된다.

5. 다음 라운드에서 root cause 판별 기준

- raw height가 계속 증가한다
  - renderer 원인 우세
  - `sendHeight.measuredHeight` 수열과 `source` 반복 패턴 확인

- raw는 거의 같은데 wrapper가 계속 적용한다
  - wrapper stale apply 또는 debounce/apply 중복 문제 우세
  - `sessionId`, `height.defer`, `height.apply`, `stale_session` 확인

- 특정 source만 반복된다
  - 예: `ResizeObserver`
  - 그 source가 다음 수정 대상

- old session apply가 섞인다
  - wrapper stale session 문제 확정
  - `height.ignore { reason: "stale_session" }` 여부 확인
