1. 이번 라운드에서 가장 가능성 높게 본 실제 원인

- wrapper가 상태 전환마다 iframe을 완전히 죽이지 못하고 있었다.
- 그 결과 `READY`가 아닌 상태에서도 기존 iframe 문서가 살아 있으면서 `RCL_REPORT_HEIGHT`를 계속 보낼 수 있었고, wrapper가 이를 다시 반영하면서 height loop처럼 보일 가능성이 컸다.
- polling도 `PROCESSING` 전용으로 강하게 묶여 있지 않아, state transition / fetch / iframe reload가 서로 겹치면 반복 렌더처럼 보일 수 있었다.

2. wrapper에서 확인한 문제점

- 상태별 공통 정리 루틴이 없어서 `showLoading`, `showState`, `showError`, `showFrame`가 각각 비슷한 정리를 따로 하고 있었다.
- `resetFrame()`은 `src` 제거만 했지, 기존 iframe 문서를 DOM 레벨에서 새로 교체하지 않았다.
- `message`는 `READY`에서만 받도록 되어 있었지만, stale debounce/timer 정리가 약했다.
- `boot()` 중복 진입을 막는 플래그가 없어서 polling이나 재호출이 겹치면 entry fetch가 중복될 수 있었다.
- 같은 token / 같은 READY 상태 재진입 시 불필요한 iframe 재주입 방어가 부족했다.

3. 수정한 항목 목록

- 파일: `imweb/premium_report.html`
- 수정 내용
  - 상태 머신 정리: `BOOT / PROCESSING / READY / STATE / ERROR`
  - 공통 상태 전환 함수 `prepareState(...)` 추가
  - 공통 정리 항목 추가
    - polling timer 정리
    - height RAF 정리
    - height debounce timer 정리
    - iframe loaded class 제거
    - iframe 기본 높이 복원
    - active token 초기화
  - iframe 완전 해제를 위해 `recreateFrame(...)` 도입
  - `showFrame(token)`만 iframe을 살릴 수 있게 변경
  - `READY`에서만 height message 반영
  - `entryInFlight`로 entry fetch 재진입 차단
  - `scheduleRecheck()`를 `PROCESSING` 전용으로 고정
  - 동일 token + 동일 READY + 동일 src 상태면 iframe 재주입 생략
  - 콘솔 디버그 로그 추가

- 파일: `app/services/reporting/premium_renderer.py`
  - 추가 수정 없음
  - 현재 debounce / duplicate-height 방지 구조 유지

4. polling / iframe / state / message 제어 구조

- 상태 전환
  - 모든 화면 전환은 `prepareState(nextState, options)`를 먼저 통과한다.
  - 여기서 polling과 pending height 작업을 정리하고, 필요 시 iframe을 완전히 재생성한다.

- iframe 제어
  - `showFrame(token)`만 iframe `src`를 주입한다.
  - `PROCESSING / STATE / ERROR`로 가면 iframe은 재생성되어 기존 문서를 끊는다.
  - `READY` + 동일 token + 동일 src면 재주입하지 않는다.

- polling 제어
  - `scheduleRecheck(token)`는 내부에서 항상 기존 timer를 먼저 제거한다.
  - `currentState !== "PROCESSING"`이면 polling 예약 자체를 거부한다.
  - `READY / STATE / ERROR / INVALID / NOT_PAID / NEED_SURVEY`에서는 polling이 남지 않는다.

- message 제어
  - `RCL_REPORT_HEIGHT`는 `currentState === "READY"`일 때만 반영한다.
  - `HEIGHT_DEBOUNCE_MS = 160`
  - `HEIGHT_MIN_DELTA = 24`
  - 작은 shrink는 `HEIGHT_SHRINK_GUARD = 96`으로 무시한다.
  - stale debounce timer도 상태 전환 시 정리한다.

5. 추가한 디버그 로그 설명

- `state.transition`
  - from -> to, reason, keepFrame 여부

- `showLoading / showState / showError / showFrame`
  - 어떤 UI 진입이 발생했는지

- `resetFrame / frame.recreate / frame.load`
  - iframe이 실제로 끊기고 다시 생성되는지

- `entry.fetch.start / entry.fetch.end / boot.resolve / boot.skip / boot.error`
  - polling loop인지, fetch 중복인지, stale response인지 확인 가능

- `scheduleRecheck / clearPolling / poll.fire`
  - polling timer가 언제 생성/삭제/발동되는지 확인 가능

- `message.receive / message.ignore / height.defer / height.apply`
  - currentState, incoming height, current height, ignore reason 확인 가능

6. 사용자가 브라우저 콘솔에서 확인해야 할 포인트

1. `NOT_PAID / NEED_SURVEY / INVALID / ERROR`
   - `state.transition` 후 `resetFrame`, `frame.recreate`가 나오는지
   - 이후 `message.ignore { reason: "state_not_ready" }`만 보이고 `height.apply`는 없어야 한다.

2. `PROCESSING`
   - `showLoading`
   - `scheduleRecheck`
   - `poll.fire`
   - `entry.fetch.start / end`
   순서만 반복되어야 하고 iframe 관련 `frame.load`는 없어야 한다.

3. `READY`
   - `showFrame`
   - `frame.load`
   - `message.receive`
   - `height.apply`
   순서가 보여야 한다.
   - 이후 작은 height 변경은 `height.ignore { reason: "below_delta" }` 또는 `shrink_guard`로 보여야 한다.

4. fetch 중복 여부
   - `boot.skip { reason: "entry_in_flight" }`가 반복적으로 많이 뜨면 외부에서 boot 재호출이 과한 것이다.

7. 남은 리스크 1~2개

- 현재 wrapper 파일의 한글 인코딩이 이미 깨진 상태라 문구 편집 시 별도 인코딩 정리가 필요할 수 있다.
- 외부 아임웹 페이지가 같은 wrapper를 중복 삽입하거나 새로고침 없이 script를 중복 실행하는 구조라면, console 로그로 실제 중복 mount 여부를 추가 확인해야 한다.
