1. raw incoming height가 증가하는지 여부

- 현재 코드 수정만으로는 브라우저 실행 전 확정 불가.
- 이번 라운드에서 wrapper/renderer 콘솔 로그를 구조화해, READY 상태에서 들어오는 `incomingRawHeight` 수열이 실제로 계속 증가하는지 바로 판별 가능하게 만들었다.
- 확인 로그
  - wrapper: `message.receive`, `message.target`, `height.apply`, `height.ignore`
  - renderer: `[RCL premium renderer] sendHeight`, `[RCL premium renderer] ignoreHeight`

2. wrapper stale apply가 있었는지 여부

- 현재 코드 수정만으로는 브라우저 실행 전 확정 불가.
- 이번 라운드에서 `frame session id`를 도입해 stale apply 여부를 직접 확인 가능하게 만들었다.
- 각 READY 진입마다 `frame.session.start`
- 비활성화 시 `frame.session.end`
- apply 시점 세션과 현재 세션이 다르면 `height.ignore { reason: "stale_session" }`

3. 가장 많이 반복된 renderer source

- 브라우저 실행 전 확정 불가.
- 대신 source별 1초 내 호출 카운터를 로그에 포함시켰다.
- 확인 대상 source
  - `DOMContentLoaded:*`
  - `load:*`
  - `fonts.ready:*`
  - `ResizeObserver`
  - `resize`
- 로그 필드
  - `perSecondCount`

4. 현재 가장 유력한 실제 원인 1개

- 가장 유력한 원인은 `ResizeObserver` 또는 `load/fonts.ready` 계열이 iframe 높이 변화에 연동되어 raw height 자체를 다시 조금씩 키우는 renderer-side feedback이다.
- wrapper stale apply 가능성도 배제하지 않았지만, 이번 라운드에서는 세션 id로 stale update를 분리해 확인 가능하게 만들었다.

5. 다음 라운드에서 수정해야 할 지점 1개

- 가장 많이 반복되는 renderer source가 확정되면, 그 source 하나를 기준으로 측정 빈도 또는 측정 기준을 더 줄이는 것이 다음 수정 포인트다.
- 우선순위는 `ResizeObserver`다.

6. 브라우저 콘솔에서 사용자가 확인할 핵심 로그 순서

1. READY 진입
   - `frame.session.start`
   - `showFrame`

2. renderer 측정
   - `[RCL premium renderer] sendHeight`
   - `source`
   - `measuredHeight`
   - `delta`
   - `selectedKey`
   - `candidates`
   - `perSecondCount`

3. wrapper 수신
   - `message.receive`
   - `sessionId`
   - `incomingRawHeight`
   - `currentAppliedHeight`
   - `delta`

4. wrapper 적용 결정
   - `message.target`
   - `height.defer`
   - `height.apply`
   - 또는 `height.ignore`

5. stale 여부 확인
   - `height.ignore { reason: "stale_session" }`
   - 이 로그가 뜨면 old iframe / old timer가 뒤늦게 반영되려던 것이다.

6. root cause 판별 기준
   - raw 자체가 계속 증가한다:
     - renderer 원인 우세
   - raw는 안정적인데 apply가 반복된다:
     - wrapper stale/debounce 원인 우세
   - 특정 source만 `perSecondCount`가 계속 높다:
     - 그 source가 1차 수정 대상
