4. Env fallback summary
- Free template
  - first: SOLAPI_TEMPLATE_FREE
  - fallback: KAKAO_ALERT_FREE_TEMPLATE_CODE
  - else: source=missing
- Premium template
  - first: SOLAPI_TEMPLATE_PREMIUM
  - fallback: KAKAO_ALERT_PREMIUM_TEMPLATE_CODE
  - else: source=missing
- Service env
  - API URL: SOLAPI_API_URL -> KAKAO_ALERT_API_URL -> https://api.solapi.com/messages/v4/send
  - API KEY: SOLAPI_API_KEY -> KAKAO_ALERT_API_KEY
  - API SECRET: SOLAPI_API_SECRET -> KAKAO_ALERT_API_SECRET
  - PFID: SOLAPI_PFID -> KAKAO_ALERT_SENDER_KEY

5. Final failure decision logic
- validation_input
  - invalid normalized phone
  - empty template_id
  - empty url_value
  - reason=INVALID_MESSAGE_INPUT
- validation_env
  - missing API KEY or API SECRET or PFID
  - reason=SOLAPI_ENV_MISSING
- response_validation
  - HTTP status >= 400
  - success == false
  - status == FAILED
  - errorCode exists
  - errorMessage exists
  - statusCode is not a success code
  - statusMessage contains FAIL / ERROR / INVALID / DENIED / UNAUTHORIZED
  - code is not a success code
  - message contains FAIL / ERROR / INVALID / DENIED / UNAUTHORIZED
- request_exception
  - request-layer exception such as httpx error

7. Most likely direct cause
- The most likely direct cause currently visible is that template_id itself is empty at runtime.
- Why this is the leading cause
  - both callers resolve template_id from env at call time
  - local .env does not define SOLAPI_TEMPLATE_FREE / SOLAPI_TEMPLATE_PREMIUM
  - local .env does not define KAKAO_ALERT_FREE_TEMPLATE_CODE / KAKAO_ALERT_PREMIUM_TEMPLATE_CODE
  - current process env also does not expose those keys
  - previous implementation did not log template source or raw response details clearly enough

- Conclusion:
- Direct cause:
  The strongest currently confirmed cause is an empty runtime template_id for free and premium alert sends. The code now logs template resolution before send and logs INVALID_MESSAGE_INPUT with template_source when template_id is blank.
- Secondary cause:
  The old implementation only kept a shallow response summary, so auth failure, template mismatch, and Solapi response errors were hard to distinguish.
